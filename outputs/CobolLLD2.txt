# Technical Document and Low-Level Design (LLD)

## 1. Introduction  
This document details the technical changes and low-level design (LLD2) for the COBOL portfolio transaction processing system, specifically regarding the introduction of a new `TRANSACTION-TYPE` field. The changes span the transaction record copybook (`TRNREC.cpy`) and the main transaction processing program (`PORTTRAN.cbl`). The purpose of this document is to provide a clear understanding of the modifications, their rationale, and the precise logic and insertion points for maintainers and future enhancements.

## 2. Existing Logic and Flow  
### 2.1 Overview  
The portfolio transaction processing system manages investment transactions, including buying, selling, transferring, and applying fees to portfolio accounts. The system reads transaction records, validates their contents, updates portfolio positions, and logs audit information.

### 2.2 Detailed Logic  
- **Initialization:**  
  - Opens transaction and portfolio files.
  - Initializes counters and file status flags.
- **Processing:**  
  - Reads each transaction record sequentially.
  - Validates transaction fields (type, amount, portfolio ID, etc.).
  - Updates portfolio positions based on transaction type.
  - Logs audit information for each processed transaction.
- **Termination:**  
  - Closes files and displays summary statistics.

#### Flowchart:  
```mermaid

    Start([Start])
    Init([1000-INITIALIZE])
    CheckStatus([Check File Status])
    ProcessLoop([2000-PROCESS-TRANSACTIONS])
```

## 3. Impacted Changes  
### 3.1 User Story or Analysis Report Summary  
**User Story:**  
As a business analyst, I require the ability to categorize transactions with an extended type (`TRANSACTION-TYPE`) for batch processing and reporting. This enables more granular reporting and control over transaction processing.

### 3.2 Impacted Code Changes Summary:

#### 3.2.1 Impacted Sections and Files:
- **src/copybook/common/TRNREC.cpy**  
  - **Purpose of Changes:**  
    - Added a new field `TRANSACTION-TYPE` (PIC X(04)) to the transaction record structure to support batch processing and reporting.
    - Adjusted the filler size to accommodate the new field.
  - **Impact:**  
    - All programs using this copybook now have access to the new field, enabling enhanced transaction categorization.

- **src/programs/portfolio/PORTTRAN.cbl**  
  - **Purpose of Changes:**  
    - Integrated validation and processing logic for the new `TRANSACTION-TYPE` field throughout the transaction lifecycle.
    - Enhanced audit logging to include the new field.
  - **Impact:**  
    - Ensures that all transactions have a valid `TRANSACTION-TYPE`.
    - Enables future business logic extensions based on this field.
    - Improves audit trail completeness.

### 3.3 Insertion Points  
- **TRNREC.cpy:**  
  - Inserted `TRANSACTION-TYPE` as the last field in the `TRN-DATA` group (after `TRN-STATUS`).
  - Reduced `TRN-FILLER` by 4 bytes to maintain record length.

- **PORTTRAN.cbl:**  
  - **WORKING-STORAGE SECTION:**  
    - Added `WS-TRANSACTION-TYPE-VLD` for validation.
  - **2100-VALIDATE-TRANSACTION:**  
    - Inserted validation to ensure `TRANSACTION-TYPE` is not spaces.
    - Moved value to working storage for further checks.
  - **2120-CHECK-TRANSACTION-TYPE:**  
    - Enhanced to validate both `TRN-TYPE` and `TRANSACTION-TYPE` against allowed values.
  - **2200-UPDATE-POSITIONS:**  
    - Comments added for future logic based on `TRANSACTION-TYPE`.
  - **2210-PROCESS-BUY, 2220-PROCESS-SELL, 2230-PROCESS-TRANSFER, 2240-PROCESS-FEE:**  
    - Comments for optional logging or logic based on `TRANSACTION-TYPE`.
  - **2300-UPDATE-AUDIT-TRAIL:**  
    - Appended `TRANSACTION-TYPE` to audit messages if present.
